## Тестування та відлагодження

Перед тим, як почати тестування вашого MCP сервера, важливо зрозуміти доступні інструменти та найкращі практики для відлагодження. Ефективне тестування забезпечує очікувану роботу сервера та допомагає швидко виявляти і вирішувати проблеми. У наступному розділі наведено рекомендовані підходи для валідації вашої реалізації MCP.

## Огляд

У цьому уроці розглядається, як обрати правильний підхід до тестування та найефективніший інструмент для тестування.

## Навчальні цілі

До кінця уроку ви зможете:

- Описати різні підходи до тестування.
- Використовувати різні інструменти для ефективного тестування вашого коду.


## Тестування MCP серверів

MCP надає інструменти для допомоги у тестуванні та відлагодженні ваших серверів:

- **MCP Inspector**: інструмент командного рядка, який можна запускати як CLI інструмент, так і у вигляді візуального інтерфейсу.
- **Ручне тестування**: Ви можете використовувати інструмент на кшталт curl для запуску веб-запитів, але підійде будь-який інструмент, здатний виконувати HTTP запити.
- **Модульне тестування**: Можна використовувати улюблену тестову рамку для тестування функцій як сервера, так і клієнта.

### Використання MCP Inspector

Ми описували використання цього інструмента у попередніх уроках, але давайте розглянемо його на загальному рівні. Це інструмент, створений на Node.js, і ви можете використати його, викликавши виконуваний файл `npx`, який тимчасово завантажить та встановить сам інструмент, після чого очистить себе після завершення вашого запиту.

[MCP Inspector](https://github.com/modelcontextprotocol/inspector) допомагає вам:

- **Виявляти можливості сервера**: автоматично визначати доступні ресурси, інструменти та підказки
- **Тестувати виконання інструментів**: пробувати різні параметри та бачити відповіді у режимі реального часу
- **Переглядати метадані сервера**: переглядати інформацію про сервер, схеми та конфігурації

Типовий запуск інструменту виглядає так:

```bash
npx @modelcontextprotocol/inspector node build/index.js
```

Вказана команда запускає MCP та його візуальний інтерфейс і відкриває локальний веб-інтерфейс у вашому браузері. Ви побачите панель керування, на якій відображаються зареєстровані MCP сервери, їхні доступні інструменти, ресурси та підказки. Інтерфейс дозволяє інтерактивно тестувати виконання інструментів, переглядати метадані сервера та бачити відповіді в реальному часі, що полегшує валідацію та відлагодження реалізацій вашого MCP сервера.

Ось як це може виглядати: ![Inspector](../../../../translated_images/uk/connect.141db0b2bd05f096.webp)

Цей інструмент також можна запускати у режимі CLI, для цього додайте атрибут `--cli`. Ось приклад запуску інструмента у режимі "CLI", який виводить усі інструменти на сервері:

```sh
npx @modelcontextprotocol/inspector --cli node build/index.js --method tools/list
```

### Ручне тестування

Окрім запуску інспектора для тестування можливостей сервера, можна використовувати схожий підхід – запускати клієнт, здатний використовувати HTTP, наприклад curl.

За допомогою curl ви можете тестувати MCP сервери безпосередньо за допомогою HTTP-запитів:

```bash
# Приклад: Метадані тестового сервера
curl http://localhost:3000/v1/metadata

# Приклад: Виконати інструмент
curl -X POST http://localhost:3000/v1/tools/execute \
  -H "Content-Type: application/json" \
  -d '{"name": "calculator", "parameters": {"expression": "2+2"}}'
```

Як видно з прикладу використання curl, ви використовуєте POST-запит для виклику інструмента з корисним навантаженням, що складається з назви інструменту та його параметрів. Використовуйте підхід, який найкраще підходить вам. CLI інструменти взагалі зазвичай швидші у використанні і дозволяють автоматизувати сценарії, що може бути корисним в середовищі CI/CD.

### Модульне тестування

Створюйте модульні тести для своїх інструментів та ресурсів, щоб переконатися, що вони працюють відповідно до очікувань. Ось приклад коду для тестування.

```python
import pytest

from mcp.server.fastmcp import FastMCP
from mcp.shared.memory import (
    create_connected_server_and_client_session as create_session,
)

# Позначте весь модуль для асинхронних тестів
pytestmark = pytest.mark.anyio


async def test_list_tools_cursor_parameter():
    """Test that the cursor parameter is accepted for list_tools.

    Note: FastMCP doesn't currently implement pagination, so this test
    only verifies that the cursor parameter is accepted by the client.
    """

 server = FastMCP("test")

    # Створіть пару інструментів для тестування
    @server.tool(name="test_tool_1")
    async def test_tool_1() -> str:
        """First test tool"""
        return "Result 1"

    @server.tool(name="test_tool_2")
    async def test_tool_2() -> str:
        """Second test tool"""
        return "Result 2"

    async with create_session(server._mcp_server) as client_session:
        # Тест без параметра cursor (пропущено)
        result1 = await client_session.list_tools()
        assert len(result1.tools) == 2

        # Тест з cursor=None
        result2 = await client_session.list_tools(cursor=None)
        assert len(result2.tools) == 2

        # Тест з cursor як рядок
        result3 = await client_session.list_tools(cursor="some_cursor_value")
        assert len(result3.tools) == 2

        # Тест з порожнім рядком cursor
        result4 = await client_session.list_tools(cursor="")
        assert len(result4.tools) == 2
    
```

Вищенаведений код виконує наступне:

- Використовує фреймворк pytest, який дозволяє створювати тести як функції та використовувати оператори assert.
- Створює MCP сервер з двома різними інструментами.
- Використовує оператор `assert` для перевірки виконання певних умов.

Подивіться на [повний файл тут](https://github.com/modelcontextprotocol/python-sdk/blob/main/tests/client/test_list_methods_cursor.py)

Маючи цей файл, ви можете протестувати власний сервер, щоб переконатися, що можливості створюються правильно.

Усі основні SDK мають подібні розділи для тестування, тому ви можете адаптувати їх до обраного середовища виконання.
